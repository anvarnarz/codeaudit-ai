🔴2a. Security — Critical Findings
🚨
Immediate action required. These 4 findings represent active security risks. The production API token is publicly visible to every website visitor, and all production secrets have been committed to version control since February 2023.
Critical Production & Development .env Files Committed to Git Secrets ▶
Files
.env.production (line 15: ASTRO_TOKEN)
.env.development (line 19: ASTRO_TOKEN as JWT)
.env.production (line 14: PUBLIC_MIXPANEL_TOKEN)
.env.production (line 13: VITE_RECAPTCHA_V3_KEY)
.env.production (line 12: PUBLIC_GTM_ID)
Finding
Both .env.production and .env.development are committed to version control and have been since commit 99303393 in February 2023. The .gitignore only excludes .env.local, not the production or development env files. All production credentials — API tokens, reCAPTCHA keys, GTM IDs, Mixpanel tokens — are readable by anyone with repository access.
Attack Vector
Any team member, contractor, or third party with read access to the repository has all production credentials. If the repository is ever made public or leaked, all credentials are immediately compromised. Even after removal from the current HEAD, secrets persist in git history forever unless history is rewritten.
Remediation
1. Rotate ALL secrets immediately (ASTRO_TOKEN, Mixpanel, reCAPTCHA, GTM)
2. Add to .gitignore: .env.production, .env.development, .env.staging
3. Run: git filter-repo --invert-paths --path .env.production --path .env.development
OR treat the repository as permanently compromised
4. Move secrets to GitHub Actions secrets or a secrets manager (Vault, AWS Secrets Manager)
Critical ASTRO_TOKEN Compiled Into Client-Side JavaScript Bundle Secrets · Auth ▶
File
src/utils/request.ts line 8
src/components/practicum/PracticumPricing.astro lines 219–222 (client import)
src/components/course/CoursePricing.astro (client import)
src/components/blog/BlogCategories.astro (client import)
Finding
request.ts sets 'x-astro-token': String(import.meta.env.ASTRO_TOKEN) as a header on every API request. This file is imported inside client-side <script> blocks in multiple components. In Astro/Vite, import.meta.env.ASTRO_TOKEN (without the PUBLIC_ prefix) is intended for server-only use — but because the file is bundled into browser JavaScript, the token value is inlined into the client bundle and is visible to every visitor via browser DevTools → Sources or by inspecting the JS bundle.
Attack Vector
Any visitor opens DevTools → Network tab, copies the x-astro-token header value from any API request, and can then make arbitrary authenticated API calls to the backend as the landing application. This bypasses any API-level access controls that rely on this token, including access to internal endpoints not intended for public use.
Remediation
1. Create two separate API clients:
- server-request.ts (used in .astro frontmatter, includes ASTRO_TOKEN)
- client-request.ts (used in <script> blocks, no token — auth via cookie only)
2. Move any requests requiring ASTRO_TOKEN to Astro's server-side data fetching
3. Rotate ASTRO_TOKEN after fix
Critical Password Field Explicitly Requested in Frontend User Profile Fetch Data Protection ▶
File
src/repositories/authRepository.ts line 13
Finding
The userSelect array includes 'password' as a field to fetch from the /users/{id} endpoint. This runs on every authenticated page load via client-side JavaScript. If the backend honors this field selector, the password hash (or in a worst case, plaintext password) is transmitted to the browser and stored in JavaScript memory where it is accessible to any XSS payload, malicious browser extension, or network observer.
Attack Vector
An attacker with any XSS vector (e.g., via the course.description innerHTML vulnerability) can read the password value from the API response or from the nanostores user state and exfiltrate it. Even without XSS, the password is visible in the browser's Network tab to anyone with physical access to the machine.
Remediation
Remove from userSelect array in authRepository.ts:
- 'password'
- 'pinfl' (Uzbek national ID — equivalent to SSN)
- 'resetId' (password reset token — allows account takeover via XSS)
- 'telegramChatId' (unnecessary PII)
Critical Public Coupon Endpoint Exposes User PII (userPhones, usersWhoUsed) Data Protection · API Security ▶
File
src/repositories/couponRepository.ts lines 13–29 (publicCouponFields array)
Finding
The /coupons/public endpoint is called with a select field list that includes userPhones, usersWhoUsed, userIds, creatorId, creator, comments, usageLimit, perUserUsageLimit, and affiliate. This is far more data than a UI requires to display a coupon discount. The endpoint is unauthenticated — any visitor can fetch it.
Attack Vector
Any unauthenticated user opens the browser network tab and reads: (1) a list of phone numbers associated with restricted coupons — PII leak; (2) the number of users who have claimed a coupon — competitive intelligence about sales volume; (3) affiliate relationships — internal business metadata. A scripted attacker can enumerate all coupons and harvest this data in bulk.
Remediation
Remove from publicCouponFields: userPhones, usersWhoUsed, userIds,
creatorId, creator, comments, affiliate, comment

The UI only needs: code, amount, amountType, expiredAt, isActive, name

Enforce field restrictions server-side regardless of client request.
🟠2b. Security — High Severity
High innerHTML XSS — course.description from API Without Sanitization Injection · XSS ▶
File
src/scripts/courses.ts line 193
cardDescription.innerHTML = course.description;
Finding
Course description from the API is inserted via innerHTML without any sanitization. If an attacker with write access to the backend CMS can inject HTML into course descriptions, arbitrary JavaScript executes on all page visitors. Combined with the password and resetId fields being fetched to the browser, an XSS attack here could harvest credentials from all logged-in users.
Remediation
Option A: Use textContent instead of innerHTML (strips all HTML)
Option B: npm install dompurify
cardDescription.innerHTML = DOMPurify.sanitize(course.description);

Also audit: set:html directives in ommaviy-oferta/*.astro and Hero2.astro
High CSP Allows unsafe-inline and unsafe-eval — XSS Protection Negated Infrastructure · CSP ▶
File
_headers — Content-Security-Policy header
script-src: ... 'unsafe-inline' 'unsafe-eval'
Finding
The CSP header is present but rendered ineffective. 'unsafe-inline' allows all inline scripts and event handlers. 'unsafe-eval' allows eval(), new Function(), and similar dynamic code execution. Together these negate XSS protection entirely — a CSP with both directives is equivalent to having no CSP.
Remediation
1. Remove 'unsafe-eval' entirely (identify and rewrite any eval usage)
2. Replace 'unsafe-inline' with nonces:
script-src 'nonce-{random}' ...
Astro supports nonce injection via middleware
3. Use strict-dynamic for third-party scripts
4. Test with: https://csp-evaluator.withgoogle.com/
High Client-Side JWT Decode Without Signature Verification Auth ▶
File
src/components/common/AuthScript.astro lines 12–18
jwtDecode(m_at) → authRepository.getUser(decodedToken._id)
Finding
The jwtDecode library decodes the JWT payload without verifying the cryptographic signature. The decoded _id is used directly to fetch user data from the API. Security depends entirely on the backend API validating the cookie token — the frontend provides no defence-in-depth. An attacker who can set the m_at cookie can potentially load any user's data on the client.
Remediation
Use Astro server-side data fetching in .astro frontmatter (runs server-side).
The cookie is available server-side; user fetch + JWT validation should happen
there, passing only non-sensitive display data to client components.
High No Package Lock File — Non-Deterministic Builds + Broken Dockerfile Infrastructure · Supply Chain ▶
Evidence
No package-lock.json, yarn.lock, or pnpm-lock.yaml in repo
.gitignore line: pnpm-lock.yaml
Dockerfile line 4: RUN npm ci ← requires package-lock.json → FAILS
Finding
Without a lock file, every npm install or pnpm install resolves to the latest compatible version of every dependency, including transitive ones. CI builds may use different package versions than production or local developer machines. A supply chain attack via a compromised transitive dependency could be silently pulled in. The Dockerfile is completely broken as a result.
Remediation
1. Remove pnpm-lock.yaml from .gitignore
2. Run: pnpm install (generates pnpm-lock.yaml)
3. Commit pnpm-lock.yaml
4. Update CI: pnpm install --frozen-lockfile
5. Update Dockerfile: RUN pnpm install --frozen-lockfile (or use COPY pnpm-lock.yaml)
High No Tests in PR Pipeline — Code Ships Without Automated Verification CI/CD · Quality ▶
File
.github/workflows/pr.yml — steps: build:dev + lint:eslint only
.github/workflows/e2e.yml — manual trigger only (schedule commented out)
Finding
The PR merge gate only builds and lints. E2E tests run on a separate daily schedule, not as a PR requirement. This means any code change — including security-sensitive areas like the payment flow, auth handling, or API layer — can be merged and deployed to production without any automated test verification.
Remediation
Add to pr.yml after the build step:
- name: Run E2E tests
run: pnpm exec playwright test --reporter=list
env:
BASE_URL: ${{ secrets.STAGING_URL }}

Set as required status check in branch protection rules.
High Errors Swallowed in Payment-Critical Code Path Error Handling · Reliability ▶
Files
src/scripts/courses.ts lines 224–226 (displayCards catch → console.log)
src/scripts/courses.ts lines 258–260 (setupCouponUpdates catch → console.log)
src/repositories/practicumRepository.ts lines 16–18 (getPricesByCoupon → return null)
Finding
Three locations in the payment and coupon flow silently swallow exceptions with console.log(error). A coupon fetch failure leaves prices displaying without the discount applied — users may believe they're getting a discount that isn't actually being processed. A course card rendering error silently empties the course list. No error is surfaced to the user or to any monitoring system.
Remediation
1. Show user-visible error states when coupon fetch fails
2. Add window.onerror and window.onunhandledrejection handlers
3. Integrate Sentry or similar for production error capture
4. Replace console.log with proper error boundary components
High Bitrix24 and GTM Scripts: Full Page Access Without SRI or Effective CSP Third-Party · Supply Chain ▶
Files
src/components/common/BasicScripts.astro lines 169–176 (Bitrix24)
src/utils/tag-manager.ts lines 7–18 (GTM-KTZ7NCL)
src/components/widgets/LeadForm.astro line 162 (imask.js from cdnjs)
Finding
Three third-party scripts load with full DOM access and no Subresource Integrity (SRI) hash. GTM allows arbitrary JavaScript injection by anyone with write access to the GTM container. Bitrix24 is served from a Russian CDN (cdn-ru.bitrix24.ru). imask.js runs on the phone number input field — a compromised CDN delivery could intercept phone numbers as users type. Combined with the ineffective CSP (unsafe-inline + unsafe-eval), there is no defence against compromised third-party scripts.
Remediation
1. Add SRI hashes to all <script> tags loading from external CDNs
2. Self-host imask.js (add to package.json)
3. Implement GTM Security Policies (restrict tag types)
4. Enable CSP nonce-based policy (see CSP finding above)
5. Review GTM container access — limit who can publish tags

src/repositories/authRepository.ts line 23 (pinfl)
