Section 02

Critical Risks

These are the issues most likely to cause a public incident, regulatory exposure, or data breach. Each requires a concrete remediation plan with a named owner and deadline.

🔑
Production Credentials Exposed in Source Code Critical

Live API keys, database passwords, and service credentials are committed directly into the codebase. Anyone with access to the repository — current or former employees, contractors, or a breach of the hosting platform — can extract these secrets and gain unauthorized access to production systems.

Business impact: Complete system compromise, regulatory fines, data breach notification obligations
👁️
API Authentication Token Visible to Every Website Visitor Critical

A private API authentication token is embedded in the client-side JavaScript bundle. This means it is downloaded and readable by every person who visits the website — including competitors, automated crawlers, and malicious actors — using standard browser developer tools. No hacking skills required.

Business impact: Unauthorized API access, service abuse, potential billing fraud, reputation risk
🗂️
Sensitive User Data Fetched to Browser High

User passwords, national identification numbers, and other personally identifiable information are being retrieved from the server and processed in the browser. This violates privacy best practices and significantly expands the attack surface. A cross-site scripting attack or malicious browser extension could silently harvest this data.

Business impact: GDPR/privacy law violation, user trust damage, potential class-action liability
⚗️
No Automated Tests Blocking Bad Code from Shipping High

The codebase has effectively zero automated test coverage. There is no safety net to catch regressions when developers make changes. Every release relies entirely on manual review. As the codebase grows and the team scales, the probability of a defect reaching production increases with each deployment.

Business impact: Increased production incident rate, slower safe release cadence, higher QA labour cost

Section 04

Investment Risk Assessment

A balanced view of risks and strengths for strategic decision-making. Security debt is the dominant concern and must be treated as a blocker.

🔴 Security Debt — Blocker Immediate

The exposed credentials and client-visible authentication tokens represent an active, exploitable vulnerability. This is not a future risk — it exists in production today. Remediation requires rotating credentials, restructuring secret management, and fixing the data exposure patterns. Estimated effort: 1–2 engineer-weeks.

🟢 Delivery Velocity — Strength Asset

65 production releases in 12 months is elite-tier performance. This indicates strong engineering discipline, a mature deployment pipeline, and a culture of continuous delivery. This velocity is a competitive advantage and suggests the team can absorb security remediation work without major slowdown.

🟢 Code Quality — Strength Asset

Low code duplication and a consistent architecture suggest the codebase is maintainable and will scale cleanly. This reduces future technical debt accumulation and means new engineers can become productive faster — lowering onboarding cost over time.

🟡 No Tests — Regression Risk Medium-term

Without automated tests, every deployment carries the risk of silently breaking existing functionality. As the user base grows, the cost of each incident grows with it. This is a compounding risk — the longer it is deferred, the more expensive it becomes to introduce tests into a mature codebase.

🟡 Bus Factor — Retention Risk Ongoing

Two engineers hold the majority of institutional knowledge. If either were to leave without knowledge transfer, delivery would slow significantly. Mitigation requires documentation, pairing practices, and intentional knowledge distribution across the team.

🔵 Architecture — Neutral Stable

The Next.js architecture is a mature, well-supported choice with a large talent pool. The current structure does not impose meaningful scaling constraints at the current growth stage. No architectural rewrite is needed in the near to medium term.

Section 05

Recommended Actions

Prioritised by business impact. The first two items are non-negotiable — they address active security vulnerabilities in production.

1
Rotate All Exposed Credentials

Immediately invalidate and replace all API keys, database passwords, and service tokens that are currently committed to the source code. Move secrets to a dedicated secret manager (e.g., environment variables via a vault service). Verify no secrets remain in git history using automated scanning tools. This resolves the two Critical findings.

Urgent — 1 day Security
2
Fix Data Exposure Vulnerabilities

Refactor API responses to ensure that sensitive fields (passwords, national IDs, private identifiers) are never included in data sent to the browser. This requires a backend code review and API audit to verify that each endpoint returns only the minimum data necessary. Implement server-side data filtering as a standard pattern going forward.

1 week Security
3
Add Automated Testing to the PR Pipeline

Introduce a minimum viable test suite covering critical user journeys (authentication, checkout, core page rendering). Configure the CI pipeline to block merges when tests fail. This does not require 100% coverage — even 40–50% coverage on critical paths dramatically reduces regression risk. Estimated effort: 2 engineer-weeks initial setup, then ongoing as a team norm.

2 weeks Quality
4
Write Project Documentation

Create and maintain a developer README covering: system architecture, environment setup, deployment process, and API contract overview. This reduces onboarding time for new engineers, mitigates bus factor risk, and is a prerequisite for scaling the team. Assign a rotating documentation owner to keep it current.

Ongoing Quality
5
Update Outdated Dependencies

Schedule a quarterly dependency review cycle. Prioritise packages flagged by automated security scanners (npm audit). Outdated dependencies with known CVEs should be treated as security issues, not housekeeping tasks. Consider adding a dependency vulnerability check to the CI pipeline to catch new issues at the point of introduction.

1–2 weeks, then quarterly Security
Bottom Line for Decision-Makers

The team is capable and delivers fast. The codebase is well-structured. The security issues are the only true blockers — they need to be fixed before any new features are shipped. Once resolved, the investment in automated testing will protect velocity as the product scales. The architecture does not need a rewrite and the team does not need to be restructured. This is a fixable situation with a clear, bounded remediation path.
